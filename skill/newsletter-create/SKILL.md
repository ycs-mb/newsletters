# Newsletter Create

Generate a new newsletter topic and scaffold everything needed to run it.

## Usage

```
/newsletter-create <topic-slug> [description]
```

Examples:
```
/newsletter-create rust-lang
/newsletter-create kubernetes "Cloud native and K8s ecosystem"
/newsletter-create indie-hacking "Bootstrapped startups, revenue milestones, launches"
```

## What This Skill Does

1. Asks clarifying questions about the topic (if not obvious from args)
2. Creates `~/newsletters/topics/<topic-slug>/` with `topic.md` and `site/template.html`
3. Runs `uv run shared/assemble_prompt.py <slug>` to compose `prompt.md` from topic.md + shared guides
4. Adds an entry to `~/newsletters/topics.toml`

> **Note:** `prompt.md` is always assembled — never hand-written. The common OUTPUT FORMAT, ACTIONS, and design rules live in `shared/prompts/design-guide.md` and `shared/prompts/ops-guide.md`.

## Step-by-Step Process

### Step 1: Parse Input

Extract from args:
- `topic-slug`: kebab-case identifier (e.g. `rust-lang`, `ai-safety`, `indie-hacking`)
- `description`: optional one-liner — if missing, ask the user

### Step 2: Gather Topic Details

Ask the user (skip if already clear from args/description):

1. **Newsletter name** — e.g. "Rust Weekly Digest", "K8s Signal"
2. **Eyebrow label** — subtitle shown above the title, e.g. "Daily Intelligence Brief"
3. **Accent color** — one of: `terracotta`, `sage`, `prussian`, `gold`
4. **Signal label** — the rating label shown in the header, e.g. "Signal", "Escalation", "Activity"
5. **Audience** — who reads this, e.g. "developers", "security researchers", "policy analysts"
6. **Primary sources** — official blogs, docs, org sites to search
7. **GitHub focus** — what repo categories to look for (frameworks, tools, libraries)
8. **Community sources** — subreddits, Discord servers, X/Twitter accounts, HuggingFace tags
9. **Section titles** — adapt the 5-section structure to the topic domain

### Step 3: Create Directory Structure

```
~/newsletters/topics/<topic-slug>/
  topic.md          # Generated by this skill
  site/
    index.html       # Will be generated on each newsletter run
    template.html    # Copied from claude-digest and accent-updated
```

```bash
mkdir -p ~/newsletters/topics/<topic-slug>/site
cp ~/newsletters/topics/claude-digest/site/template.html ~/newsletters/topics/<topic-slug>/site/template.html
```

Edit the copied template to update:
- `data-accent="terracotta"` → `data-accent="{ACCENT}"`
- Signal label text if it differs from "Signal"

### Step 4: Add to topics.toml

Append to `~/newsletters/topics.toml`:

```toml
[{TOPIC_SLUG}]
name = "{TOPIC_NAME}"
description = "{TOPIC_DESCRIPTION}"
accent = "{ACCENT}"
signal_label = "{SIGNAL_LABEL}"
folder = "topics/{TOPIC_SLUG}"
eyebrow = "{EYEBROW}"
```

### Step 5: Generate topic.md

Write `~/newsletters/topics/<topic-slug>/topic.md` using the template below.
This is the **only file you author** — it contains the topic-specific research identity.

### Step 6: Assemble prompt.md

```bash
cd ~/newsletters && uv run shared/assemble_prompt.py <topic-slug>
```

This combines `topic.md` + `shared/prompts/design-guide.md` + `shared/prompts/ops-guide.md` into the final `prompt.md`.

### Step 7: Confirm

Show the user:
- Path to `topic.md` and assembled `prompt.md`
- How to run it, either through the manage UI at `http://localhost:9000/manage.html`
  or directly with the default OpenRouter-backed helper:
  ```bash
  uv run python -c "from shared.newsletter_generation import generate_newsletter_issue; generate_newsletter_issue('{TOPIC_SLUG}')"
  ```
- The portal URL after build: `http://100.110.249.12:8787/{TOPIC_SLUG}/`

## topic.md Template

```markdown
# {TOPIC_NAME} — Topic Brief

## Identity
- Role: technical newsletter curator covering {TOPIC_DOMAIN} for {TOPIC_AUDIENCE}
- Audience: {TOPIC_AUDIENCE}
- Signal label: {SIGNAL_LABEL} (1–5, where 5 = {SIGNAL_MAX_DESCRIPTION})

## Sources

**{PRIMARY_SOURCE_CATEGORY}:**
- {SOURCE_1}: {URL_1}
- {SOURCE_2}: {URL_2}

**{SECONDARY_SOURCE_CATEGORY}:**
- {SOURCE_3}: {URL_3}

**Community:**
- {COMMUNITY_1}: {URL_OR_HANDLE}
- {COMMUNITY_2}: {URL_OR_HANDLE}

**GitHub ecosystem:**
- {GITHUB_SEARCH_TERMS}

## Sections

### Section 01: {SECTION1_TITLE} (past {TIMEFRAME})
{What to track and how. Include formatting notes e.g. wrap CLI flags in <code> tags.}
If nothing new: "{EMPTY_STATE_MESSAGE}"

### Section 02: {SECTION2_TITLE} (past {TIMEFRAME})
{What to highlight. Specify highlight card category mapping:}
`voice` ({COLOR_MEANING}), `model` ({COLOR_MEANING}), `company` ({COLOR_MEANING}), `promo` ({COLOR_MEANING}).
Aim for 4–6 cards.

### Section 03: GitHub Picks (past 7 days)
Trending or newly notable repos related to {TOPIC_DOMAIN}.
Include: name, stars, 1-line description, why it's interesting, difficulty.
Aim for 5–8 repos.

### Section 04: Community & Research (past 24 hours)
{Community sources to scan and what to surface. Include: title, engagement, 1-sentence summary, link.}

### Section 05: Tip of the Day
One practical, actionable tip for {TOPIC_AUDIENCE}.
Include a code example if applicable. Prefer tips that are non-obvious or recently enabled.
```

## Section 01 Adaptation Guide

Section 01 uses `.release-item` (version timeline). For topics without versioned releases, adapt:

| Topic Type | Section 01 Content |
|---|---|
| Software (Rust, K8s) | Version releases, changelogs, RFCs |
| Company/Product | Feature launches, pricing changes, partnerships |
| Research field | New papers, preprints, conference talks |
| Industry/Market | News, regulations, funding rounds |

The `.release-item` grid works for any dated entries — just use the date column and description.
